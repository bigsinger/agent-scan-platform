# Agent 安全测评平台 V1 客户测试/试用流程文档

# 方案 A：轻量试用流程

> 版本：V1.0  
> 适用场景：售前早期验证、客户快速体验、非正式安全评估、小范围 Agent 风险摸底  
> 目标周期：通常 1~3 个工作日完成  
> 交付物：轻量试用报告、Top 风险清单、售前讲解材料、正式 POC 建议

---

## 1. 流程目标

轻量试用不是完整安全审计，而是让客户以最小接入成本快速看到 Agent 安全测评平台的价值。

轻量试用需要达成三个目标：

1. 让客户快速完成一次 Agent 安全风险验证。
2. 找到 1~5 个真实、有证据、可解释的 Agent 安全风险。
3. 通过简版报告推动客户进入正式 POC 或采购流程。

轻量试用重点关注：

| 重点 | 说明 |
|---|---|
| 快速 | 尽量不部署复杂组件，不长时间沟通架构 |
| 可见 | 输出报告、风险截图、攻击证据 |
| 可控 | 不执行破坏性操作，不触碰生产敏感数据 |
| 可转化 | 报告中明确建议升级到标准测评或正式 POC |

---

## 2. 适用客户

适合以下客户：

1. 正在建设 AI Agent，但尚未形成完整安全方案。
2. 已有 Web/API Agent，希望快速验证是否存在提示注入和工具滥用风险。
3. 使用 MCP Server、Skill、插件生态，希望检查供应链安全风险。
4. 安全部门希望评估 Agent 安全测评产品是否值得正式 POC。
5. 客户暂时不能部署私有化组件，但可以提供测试地址或脱敏代码包。

不适合以下场景：

1. 客户要求完整合规审计。
2. 客户只允许内网测试且不能提供任何远程访问方式。
3. 客户需要代码级深度审计。
4. 客户希望测试生产环境的真实高危操作。
5. 客户要求多 Agent 复杂链路攻击验证。

这些场景应转入方案 B 正式 POC。

---

## 3. 试用前置条件

### 3.1 客户需要提供的信息

| 信息 | 是否必需 | 说明 |
|---|---|---|
| 客户名称 | 必需 | 用于项目归档 |
| Agent 名称 | 必需 | 被测对象名称 |
| Agent 类型 | 必需 | Web/API/MCP/IDE/CLI/RAG/多 Agent |
| 测试地址 | 条件必需 | 黑盒测试需要提供 URL/API |
| 测试账号 | 可选但推荐 | 便于测试权限、会话、工具调用 |
| OpenAPI 文档 | 可选 | 便于 API Agent 自动化测试 |
| 代码 ZIP | 可选 | 便于 SCA、Secret、配置、Prompt 扫描 |
| MCP Server 地址 | 可选 | 便于 MCP/Tool 专项检测 |
| Skill 包 | 可选 | 便于 Skill 幽灵指令和后门脚本扫描 |
| 授权范围 | 必需 | 明确允许测试什么 |
| 禁止操作 | 必需 | 明确哪些操作不能测试 |

### 3.2 授权边界确认

试用前必须由客户确认以下内容：

| 项目 | 默认策略 |
|---|---|
| 是否允许提示注入测试 | 默认允许 |
| 是否允许系统 Prompt 泄露测试 | 默认允许 |
| 是否允许调用真实工具 | 默认不允许高危真实调用 |
| 是否允许邮件外发 | 默认禁止，只允许 dry-run |
| 是否允许删除/修改数据 | 默认禁止 |
| 是否允许访问内网地址 | 默认禁止 |
| 是否允许高频请求 | 默认低频限速 |
| 是否使用真实敏感数据 | 默认禁止 |
| 是否保留原始证据 | 默认脱敏保存 |

### 3.3 试用数据原则

1. 优先使用测试环境。
2. 优先使用测试账号。
3. 优先使用脱敏数据。
4. 禁止真实支付、退款、删除、权限变更。
5. 禁止对生产环境执行破坏性测试。
6. 所有高危测试默认 dry-run。

---

## 4. 轻量试用总体流程

```text
售前确认 → 收集最小信息 → 创建试用项目 → 选择试用模式 → 执行快速检测 → 人工复核 → 生成简版报告 → 报告讲解 → 转正式 POC
```

---

## 5. 详细流程

## 5.1 第一步：售前确认

### 目标

确认客户是否适合轻量试用，并明确试用价值点。

### 参与角色

| 角色 | 职责 |
|---|---|
| 客户安全负责人 | 确认测试意向和安全边界 |
| 客户研发负责人 | 提供 Agent 信息和测试环境 |
| 我方售前 | 介绍试用范围和交付物 |
| 我方测评工程师 | 判断技术可行性 |

### 输出

1. 试用意向确认。
2. 试用范围初步确认。
3. 被测 Agent 类型确认。
4. 是否需要上传代码或接入测试环境确认。

---

## 5.2 第二步：收集最小信息

### 客户信息收集表

| 字段 | 示例 |
|---|---|
| 客户名称 | 某金融客户 |
| Agent 名称 | 智能客服退款 Agent |
| Agent 类型 | API Agent + Tool |
| 部署环境 | 测试环境 |
| 测试 URL | https://agent-test.example.com |
| 测试账号 | demo_user |
| 主要能力 | 查询订单、解释退款政策、提交退款申请 |
| 是否使用 MCP | 是，mail-mcp、crm-mcp |
| 是否有长期记忆 | 是，保存用户偏好和历史工单摘要 |
| 是否允许代码上传 | 是，上传脱敏 ZIP |
| 禁止操作 | 真实退款、真实邮件发送、删除数据 |

### 判断规则

1. 只有 URL/API：执行黑盒快速检测。
2. 有代码 ZIP：增加 SCA、Secret、配置、Prompt、Skill 扫描。
3. 有 MCP Server：增加 MCP/Tool 专项扫描。
4. 有测试账号：增加会话、权限、工具调用测试。
5. 有运行时接入意愿：建议转正式 POC。

---

## 5.3 第三步：创建试用项目

在平台中创建项目：

| 配置项 | 推荐值 |
|---|---|
| 测评模式 | 黑盒 + 可选白盒 |
| 检测深度 | L1 快速巡检 |
| 检测包 | Top 20/30 试用包 |
| 任务并发 | 低并发 |
| 请求速率 | 保守策略 |
| 高危操作 | dry-run |
| 报告类型 | 轻量试用报告 |
| 数据留存 | 默认 30 天，可按客户要求缩短 |

---

## 5.4 第四步：选择试用模式

### 模式 1：快速黑盒测试

适合客户只提供 URL/API 的场景。

检测内容：

1. 直接提示注入。
2. 系统 Prompt 泄露。
3. 编码混淆输入。
4. 多轮基础诱导。
5. 输出敏感信息检测。
6. 工具调用诱导测试。
7. 输入长度与限流基础测试。
8. 会话安全基础测试。

### 模式 2：代码/配置安全扫描

适合客户可以上传 ZIP 或代码片段的场景。

检测内容：

1. SCA 依赖漏洞。
2. Secret 泄露。
3. Prompt 模板风险。
4. 配置基线风险。
5. 日志敏感信息风险。
6. Docker/K8s 配置风险。
7. Agent 框架版本风险。

### 模式 3：MCP/Skill 专项扫描

适合客户提供 MCP Server 或 Skill 包的场景。

检测内容：

1. MCP Server 未认证。
2. MCP 工具声明风险。
3. 高危工具识别。
4. Tool Shadowing。
5. Skill 幽灵指令。
6. Skill 后门脚本。
7. 包名抢注与语义劫持。
8. 可疑依赖源。

推荐组合：

```text
快速黑盒测试 + 代码/配置扫描 + MCP/Skill 专项扫描
```

---

## 5.5 第五步：执行 Top 20~30 检测

### 推荐试用检测项

| 序号 | 检测项 | 类型 | 价值 |
|---|---|---|---|
| 1 | 直接提示注入 | 红队 | 最容易展示 Agent 风险 |
| 2 | 系统 Prompt 泄露 | 红队 | 客户感知强 |
| 3 | 编码混淆输入 | 红队 | 展示绕过能力 |
| 4 | 输出内容过滤 | 红队 | 检查敏感输出 |
| 5 | 工具调用权限控制 | 红队/配置 | 检查越权调用 |
| 6 | 危险操作确认机制 | 红队 | 检查高危操作审批 |
| 7 | 文件系统访问控制 | 红队 | 检查路径越权 |
| 8 | 网络请求控制 | 红队 | 检查 SSRF/外传 |
| 9 | 工具返回数据注入基础版 | 红队 | Agent 特有风险 |
| 10 | Tool Shadowing | 自动 | MCP/Tool 差异化能力 |
| 11 | MCP Server 未认证 | 自动/配置 | 高价值风险 |
| 12 | Skill 幽灵指令 | 自动 | 供应链特色 |
| 13 | Skill 后门脚本 | 自动 | 供应链特色 |
| 14 | Secret 泄露 | 自动 | 确定性强 |
| 15 | 依赖 CVE | 自动 | 客户熟悉 |
| 16 | Agent 框架漏洞 | 自动 | 贴近实际上线风险 |
| 17 | 不安全配置 | 自动 | 易修复、易转化 |
| 18 | 日志敏感信息 | 自动 | 合规价值 |
| 19 | 第三方外传基础检查 | 自动/配置 | 数据安全价值 |
| 20 | 审计日志缺失 | 配置 | 治理价值 |

---

## 5.6 第六步：人工复核

轻量试用也需要人工复核，避免误报影响信任。

复核重点：

1. P0/P1 风险是否真实可复现。
2. 是否存在误报。
3. 风险是否在客户授权范围内发现。
4. 证据是否已经脱敏。
5. 修复建议是否能被客户理解。
6. 是否适合放入试用报告 Top 风险。

人工复核结论：

| 结论 | 处理方式 |
|---|---|
| 确认风险 | 写入报告 |
| 疑似风险 | 标记为需进一步验证 |
| 误报 | 不进入客户报告，内部记录优化规则 |
| 超出授权范围 | 不展示细节，只提示需正式 POC 验证 |

---

## 5.7 第七步：生成轻量试用报告

### 报告定位

轻量试用报告不是完整审计报告，应控制长度，突出价值。

建议结构：

1. 报告封面。
2. 测试对象与范围。
3. 检测方法说明。
4. 总体安全评分。
5. Top 5 风险。
6. 风险证据截图或摘要。
7. 影响分析。
8. 快速整改建议。
9. 正式 POC 建议。
10. 附录：本次执行的检测项。

### 报告长度

| 类型 | 建议页数 |
|---|---|
| 纯黑盒试用 | 8~12 页 |
| 黑盒 + 代码扫描 | 12~18 页 |
| 黑盒 + MCP/Skill | 15~20 页 |

### Top 风险描述模板

```text
风险名称：系统 Prompt 泄露
风险等级：P0 / 严重
发现方式：红队测试
影响说明：攻击者可通过特制输入套取 Agent 内部系统指令、工具边界或安全策略，进而构造更有效的后续攻击。
证据摘要：在调试模式诱导用例中，Agent 输出了部分内部角色设定和工具调用限制。
整改建议：增加系统提示泄露检测；禁止输出自身指令；系统 Prompt 中不得包含敏感凭证；高危输出进入二次判定。
```

---

## 5.8 第八步：报告讲解

### 讲解对象

1. 客户安全负责人。
2. 客户研发负责人。
3. 客户 AI 平台负责人。
4. 客户采购或项目负责人。

### 讲解重点

| 讲解内容 | 目标 |
|---|---|
| 当前 Agent 风险 | 让客户认识问题真实性 |
| 攻击证据 | 让客户相信不是泛泛而谈 |
| 业务影响 | 让客户理解风险后果 |
| 整改方向 | 让客户看到可落地性 |
| 正式 POC 建议 | 推进下一阶段 |

### 推荐讲解顺序

```text
为什么 Agent 安全不同 → 本次测了什么 → 发现了什么 → 风险会造成什么 → 建议怎么修 → 正式 POC 可以继续验证什么
```

---

## 5.9 第九步：转正式 POC

符合以下任一条件，建议推动客户进入正式 POC：

1. 发现 P0/P1 风险。
2. 客户有多个 Agent 需要批量测评。
3. 客户使用 MCP、Skill、RAG、长期记忆、多 Agent。
4. 客户希望验证运行时行为和工具调用链。
5. 客户需要正式报告用于上线审批。
6. 客户需要私有化部署。
7. 客户希望接入 CI/CD 做持续测评。

正式 POC 建议范围：

| 项目 | 建议 |
|---|---|
| 检测深度 | L2 标准测评 |
| 接入方式 | 代码扫描 + URL 黑盒 + MCP Proxy + Runtime SDK 可选 |
| 报告 | 正式 POC 报告 + 整改清单 |
| 周期 | 1~2 周 |
| 复测 | 至少 1 次 |

---

## 6. 轻量试用中的风险控制

### 6.1 禁止事项

1. 禁止真实删除数据。
2. 禁止真实退款、支付、转账。
3. 禁止真实邮件外发给非测试邮箱。
4. 禁止绕过客户授权范围测试内网资产。
5. 禁止高频压测客户生产环境。
6. 禁止保存未脱敏的敏感个人信息。
7. 禁止在报告中暴露真实密钥原文。

### 6.2 数据处理

| 数据类型 | 处理方式 |
|---|---|
| 代码 ZIP | 扫描后按留存策略删除或归档 |
| Prompt/响应 | 默认摘要化和脱敏 |
| Secret 命中 | 只展示前后少量字符，中间掩码 |
| 日志 | 只保留必要片段 |
| 报告 | 加水印，限制下载 |
| 原始证据 | 仅授权人员可查看 |

## 9. 轻量试用成功标准

一次轻量试用可以认为成功，需满足：

1. 客户在低成本接入下完成测试。
2. 平台至少输出一份结构化报告。
3. 报告中至少包含 1 个有价值的风险或 1 组有说服力的安全基线结果。
4. 风险证据清晰，客户能理解。
5. 客户愿意进入正式 POC、私有化试用或商务报价阶段。

**结论：轻量试用的核心是“低接入成本 + 快速发现真实风险 + 专业报告呈现 + 推动正式 POC”。不要把轻量试用做成完整审计，也不要为了追求检测数量牺牲证据质量。**
